# Usage Guide — Multi-Agent Orchestration System

This is the practical reference for working with the Claude Code multi-agent orchestration framework. It covers the generic infrastructure (Part 1) that any project can use. For project-specific conventions, see the project's own guide (e.g., `{{PROJECT_ID}}-guide.md`).

---

## Quick Reference

### Routing Cheat Sheet

```
I need to...                          → Use this
─────────────────────────────────────────────────────────────
Read 1-2 files                        → Read tool
Read 3+ files                         → mcp__gemini__analyze_files
Review a diff < 50 lines              → Inline review
Review a diff 50+ lines               → mcp__gemini__review_diff
Orient to the project / "how does X"  → mcp__gemini__explain_architecture
Reason without tool use               → mcp__gemini__ask_gemini
Rebuild codebase digest               → mcp__gemini__refresh_index
Recall project context                → mcp__mem0__search_memories
Store a decision or pattern           → mcp__mem0__add_memory
Check model availability              → mcp__orchestrator__get_quota_state
Run a multi-step task                 → mcp__orchestrator__run_workflow
Work through a design problem         → sequential-thinking MCP
Quick fix / boilerplate               → Agent tool (model: haiku)
Multi-file implementation             → Agent tool (model: sonnet)
Final decision / ambiguous debug      → Handle directly (Opus)
```

### Model Cost Ranking (cheapest first)

| Tier | Model | Cost | Use For |
|------|-------|------|---------|
| FREE | Gemini 2.5 Flash Lite | $0 | Indexing, memory extraction, classification |
| FREE | Gemini 3 Flash | $0 | Reviews, analysis, brainstorming, bulk file reads |
| FREE | Gemini 3.1 Pro | $0 | Complex reasoning, architecture, long-context |
| LOW | Haiku 4.5 | $ | Quick fixes, boilerplate, simple edits |
| MED | Sonnet 4.6 | $$ | Multi-file implementation, tests, refactoring |
| HIGH | Opus 4.6 | $$$ | Final decisions, ambiguous debugging, architecture sign-off |

**Rule: Always use the cheapest model that can do the job.** Gemini handles everything it can; Claude is reserved for what requires it.

## 1. MCP Servers

### gemini-delegate — Analysis Offloading

Offloads expensive read-heavy work to Gemini's large context window at zero Claude cost.

**Tools:**

| Tool | When to Use | Example |
|------|-------------|---------|
| `analyze_files` | Reading 3+ files to answer a question | `analyze_files(["src/Models/User.cs", "src/Services/UserService.cs", "src/Data/UserRepo.cs"], "How does user creation work?")` |
| `review_diff` | Reviewing changes over 50 lines | `review_diff("git diff HEAD~3")` |
| `explain_architecture` | Session start orientation or broad "how does X work" | `explain_architecture("src/")` |
| `refresh_index` | After major code changes (new module, refactor) | `refresh_index()` |
| `ask_gemini` | General questions, summaries, brainstorming | `ask_gemini("Compare two approaches for implementing X")` |

**Pitfalls:**
- The proxy strips `cache_control` headers — no prompt caching is available for Gemini calls.
- Don't use `ask_gemini` for tasks that need follow-up tool calls — Claude should handle those directly.
- If `explain_architecture` returns stale info, run `refresh_index` first.

### mem0-mcp — Persistent Memory

Stores and retrieves project knowledge across sessions using vector search (Qdrant) and a knowledge graph (Neo4j).

**Tools:**

| Tool | When to Use |
|------|-------------|
| `search_memories` | **Always** before asking the user for project context |
| `add_memory` | After making an architectural decision, discovering a pattern, or completing a workflow |
| `search_graph` | For relational questions ("what entities relate to X?") |
| `get_entity` | To inspect a specific node in the knowledge graph |
| `list_memories` | To audit or prune stored memories |
| `delete_memory` | To remove outdated or incorrect memories |

**Protocol:**
1. At session start: `search_memories("project context")` before asking the user anything.
2. After significant work: `add_memory(...)` to store outcomes, decisions, and patterns.
3. Periodically: Review and prune stale memories.

**Pitfalls:**
- Never use the `Write` tool to modify `.claude/projects/*/MEMORY.md` — all persistence goes through mem0.
- Relationship names in memory text should use underscores, not hyphens (e.g., "has_many" not "has-many") to avoid Neo4j Cypher syntax errors.
- Embeddings are generated by Ollama (bge-m3) locally — if Ollama is down, `add_memory` will fail.

### orchestrator-mcp — Workflow Management

Manages multi-step stateful workflows via LangGraph with SQLite checkpointing.

**Tools:**

| Tool | When to Use |
|------|-------------|
| `run_workflow` | Multi-step tasks: features, refactors, reviews |
| `workflow_status` | Check progress, see if human input is needed |
| `list_workflows` | See all active/completed workflows |
| `cancel_workflow` | Abort a running workflow |
| `get_quota_state` | **Always** before spawning expensive subagents |
| `get_cost_report` | Periodically, to check spending and optimize routing |

**Generic workflow types:**

| Type | Budget | Steps |
|------|--------|-------|
| `feature` | $2.00 | plan → route → execute → review |
| `refactor` | $2.00 | analyze_scope → plan → execute_change → verify |
| `review` | $0.50 | analyze → report |
| `sprint` | $0.50 | decompose → prioritize → execute |

Projects can define additional workflow types in their `<project>-workflows` skill.

**Pitfalls:**
- Workflows pause automatically if `cost_limit` is exceeded — check `workflow_status` for `needs_human_input: true`.
- Always call `get_quota_state` before `run_workflow` to verify model availability.

---

### langfuse-mcp — Observability Bridge

Reads traces and metrics from the project's Langfuse instance.

**Tools:**

| Tool | When to Use |
|------|-------------|
| `log_event` | Record an agent action or decision manually |
| `get_cost_report` | Analyze spending by model, time period, or workflow |
| `get_agent_performance` | View agent metrics (latency, success rate) |
| `get_traces` | Get full trace for a workflow (all steps, models, costs) |

**Note:** Most logging is automatic via the `PostToolUse` hook — you rarely need to call `log_event` manually.

---

### sequential-thinking — Reflective Reasoning

A self-contained MCP for multi-step reasoning on design problems. Use it when you need to think through a complex architectural decision before acting.

## 2. Agent Roles

When delegating work via the `Agent` tool, select the appropriate role and model:

| Role | Model | Use For |
|------|-------|---------|
| **Orchestrator** | Opus 4.6 | Task decomposition, delegation (default role) |
| **Architect** | Gemini 3.1 Pro | System design, entity modeling, ADRs |
| **Implementer** | Sonnet 4.6 | Multi-file code changes, tests, refactoring |
| **Reviewer** | Gemini 3 Flash | Code review, checklist validation |
| **Fast-Coder** | Haiku 4.5 | Simple edits, boilerplate, validation code |
| **Researcher** | Gemini 3.1 Pro | Codebase exploration, log analysis |

Each project adds role-specific skills and review checklists via its `<project>-patterns` skill and CLAUDE.md `Project Agent Roles` section.

**Escalation path:** Flash → Pro → Haiku → Sonnet → Opus. If a cheaper model fails, escalate to the next tier.

**Delegation decision tree:**
```
Is it a single-step task?
  ├─ Yes → Agent tool with appropriate model
  └─ No (multi-step) → run_workflow from orchestrator-mcp

Is it a design problem?
  └─ Yes → sequential-thinking MCP first, then delegate implementation
```

## 3. Skills

Skills are markdown files that provide behavioral guidance to agents. They are loaded automatically based on context.

### Generic skills (in `~/.claude/skills/`)

| Skill | Purpose | Loaded When |
|-------|---------|-------------|
| `agent-orchestrator` | Task decomposition and role selection | Multi-step tasks |
| `tool-router` | Decision tree for optimal tool/model selection | Tool selection decisions |
| `quota-guard` | Always-active cost awareness | Every session |
| `skill-awareness` | Knowledge about the skills ecosystem | Skill-related queries |

### Project skills (in `<project>/.claude/skills/`)

Each project defines two skills:

- **`<project>-patterns`** — Domain entities, architecture conventions, API patterns, testing conventions. Loaded during any development work.
- **`<project>-workflows`** — Project-specific workflow types (e.g., new-entity, new-page). Loaded when creating new project components.

See the project-specific guide for details on these skills.

## 4. Hooks (Automatic)

These run automatically — no manual invocation needed:

| Hook | Trigger | Effect |
|------|---------|--------|
| `log-to-langfuse.py` | PostToolUse (`mcp__*`) | Logs every MCP tool call to Langfuse |
| `quota-check.py` | PreToolUse (`Agent`) | Warns/blocks before expensive Agent spawns |
| `update-task-artifact.py` | PostToolUse (`Task*`) | Renders live task list to `.claude/artifacts/tasks.md` |
| `update-workflow-artifact.py` | PostToolUse (`mcp__orchestrator__*`) | Renders workflow status to `.claude/artifacts/workflow_status.md` |

Hooks have a 3-5 second timeout and are non-blocking on failure.

### Artifact Skills (User-Invoked)

| Skill | Invocation | Artifact Generated |
|-------|------------|-------------------|
| `implementation-plan` | `/implementation-plan` | `.claude/artifacts/implementation_plan.md` |
| `walkthrough` | `/walkthrough` | `.claude/artifacts/walkthrough.md` |

See `artifacts.md` for the full artifact system documentation.

## 5. Common Workflows

### Starting a New Session
```
1. search_memories("recent work")           → Recall context
2. explain_architecture()                    → Orient to codebase
3. get_quota_state()                         → Check model availability
```

### Code Review
```
1. Get the diff (git diff, PR diff, etc.)
2. If < 50 lines → review inline
3. If 50+ lines → review_diff("the diff content")
4. Check against project-specific review checklist
```

### After Major Changes
```
1. /walkthrough                              → Generate work summary artifact
2. refresh_index()                           → Rebuild codebase digest
3. add_memory("Completed: description")      → Store in mem0
4. get_cost_report()                         → Check spending
```

### Adding a Feature (generic)
```
1. get_quota_state()                         → Verify models available
2. /implementation-plan                      → Draft plan artifact for review
3. Use sequential-thinking for design if needed
4. run_workflow("feature", "description")    → Multi-step execution (generates task_plan.md)
5. /walkthrough                              → Generate summary artifact
6. refresh_index()                           → Update digest
7. add_memory("Added feature: ...")          → Store outcome
```

For project-specific workflows (new-entity, new-page, etc.), see the project guide.

## 6. Project Customization (Part 2)

Each project extends the generic framework with:

1. **`.mcp.json`** — MCP server config with project-specific `MEM0_APP_ID` and Langfuse API keys.
2. **`.envrc`** — Environment variables (managed by direnv).
3. **`CLAUDE.md`** — Project-specific agent roles, review checklists, and development rules.
4. **`.claude/skills/<project>-patterns/`** — Domain knowledge skill.
5. **`.claude/skills/<project>-workflows/`** — Workflow definitions skill.
6. **`.claude/settings.local.json`** — MCP permissions and artifact hooks.
7. **`.claude/hooks/`** — Artifact generation hooks (`update-task-artifact.py`, `update-workflow-artifact.py`).
8. **`.claude/artifacts/`** — Generated deliverables: task plans, implementation plans, reviews, walkthroughs.
9. **`.gemini-index`** — Codebase digest (rebuilt via `refresh_index`).
10. **mem0 memories** — Seeded entity relationships and project facts.

See `project-setup-guide.md` for step-by-step instructions on creating a new project instance.

## 7. Troubleshooting

| Problem | Solution |
|---------|----------|
| MCP server not connecting | Check `uv run` path and env vars in `.mcp.json` |
| mem0 search returns nothing | Verify `MEM0_APP_ID` matches in `.envrc` and `.mcp.json` |
| Gemini calls failing | Check proxy at `localhost:1337/health` |
| Langfuse empty | Verify API keys in `.mcp.json`, check hook is registered |
| Neo4j Cypher errors | Use underscores not hyphens in relationship names |
| PointStruct validation errors | Stale entries with null vectors — non-blocking, ignore or prune |
| "ThinkingBlock" crash in mem0 | Monkey-patch is applied at server startup — restart the MCP server |
| Ollama not reachable | Start Ollama (`ollama serve`) — needed for mem0 embeddings |
