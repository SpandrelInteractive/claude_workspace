# Usage Guide ‚Äî Multi-Agent Orchestration System

This is the practical reference for working with the Claude Code multi-agent orchestration framework. It covers the generic infrastructure (Part 1) that any project can use. For project-specific conventions, see the project's own guide (e.g., `{{PROJECT_ID}}-guide.md`).

## 1. MCP Servers

### gemini-delegate ‚Äî Analysis Offloading

Offloads expensive read-heavy work to Gemini's large context window at zero Claude cost.

**Tools:**

| Tool | When to Use | Example |
|------|-------------|---------|
| `analyze_files` | Reading 3+ files to answer a question | `analyze_files(["src/Models/User.cs", "src/Services/UserService.cs", "src/Data/UserRepo.cs"], "How does user creation work?")` |
| `review_diff` | Reviewing changes over 50 lines | `review_diff("git diff HEAD~3")` |
| `explain_architecture` | Session start orientation or broad "how does X work" | `explain_architecture("src/")` |
| `refresh_index` | After major code changes (new module, refactor) | `refresh_index()` |
| `ask_gemini` | General questions, summaries, brainstorming | `ask_gemini("Compare two approaches for implementing X")` |

**Pitfalls:**
- The proxy strips `cache_control` headers ‚Äî no prompt caching is available for Gemini calls.
- Don't use `ask_gemini` for tasks that need follow-up tool calls ‚Äî Claude should handle those directly.
- If `explain_architecture` returns stale info, run `refresh_index` first.

### mem0-mcp ‚Äî Persistent Memory

Stores and retrieves project knowledge across sessions using vector search (Qdrant) and a knowledge graph (Neo4j).

**Tools:**

| Tool | When to Use |
|------|-------------|
| `search_memories` | **Always** before asking the user for project context |
| `add_memory` | After making an architectural decision, discovering a pattern, or completing a workflow |
| `search_graph` | For relational questions ("what entities relate to X?") |
| `get_entity` | To inspect a specific node in the knowledge graph |
| `list_memories` | To audit or prune stored memories |
| `delete_memory` | To remove outdated or incorrect memories |

**Protocol:**
1. At session start: `search_memories("project context")` before asking the user anything.
2. After significant work: `add_memory(...)` to store outcomes, decisions, and patterns.
3. Periodically: Review and prune stale memories.

**Pitfalls:**
- Never use the `Write` tool to modify `.claude/projects/*/MEMORY.md` ‚Äî all persistence goes through mem0.
- Relationship names in memory text should use underscores, not hyphens (e.g., "has_many" not "has-many") to avoid Neo4j Cypher syntax errors.
- Embeddings are generated by Ollama (bge-m3) locally ‚Äî if Ollama is down, `add_memory` will fail.

---

### orchestrator-mcp ‚Äî Workflow Management

Manages multi-step stateful workflows via LangGraph with SQLite checkpointing.

**Tools:**

| Tool | When to Use |
|------|-------------|
| `run_workflow` | Multi-step tasks: features, refactors, reviews |
| `workflow_status` | Check progress, see if human input is needed |
| `list_workflows` | See all active/completed workflows |
| `cancel_workflow` | Abort a running workflow |
| `get_quota_state` | **Always** before spawning expensive subagents |
| `get_quota_report` | Periodically, to check velocity, model limits, and lockout risk |

**Generic workflow types:**

| Type | Budget | Steps |
|------|--------|-------|
| `feature` | $2.00 | plan ‚Üí route ‚Üí execute ‚Üí review |
| `refactor` | $2.00 | analyze_scope ‚Üí plan ‚Üí execute_change ‚Üí verify |
| `review` | $0.50 | analyze ‚Üí report |
| `sprint` | $0.50 | decompose ‚Üí prioritize ‚Üí execute |
| `spdd_feature` | $2.00 | context acquire ‚Üí research ‚Üí spec ‚Üí synthesize (structured pipeline with verification gates) |

Projects can define additional workflow types in their `<project>-workflows` skill.

**Pitfalls:**
- Workflows pause automatically if `cost_limit` is exceeded ‚Äî check `workflow_status` for `needs_human_input: true`.
- Always call `get_quota_state` before `run_workflow` to verify model availability.

---

### langfuse-mcp ‚Äî Observability Analytics (Read-Only)

Pure analytics engine over Langfuse traces. All tracing is handled by hooks and proxy middleware ‚Äî this server only reads.

**Tools:**

| Tool | When to Use |
|------|-------------|
| `get_cost_report` | Analyze spending by source, model, or agent for a time period |
| `get_agent_performance` | View agent metrics (action breakdown, models used, avg latency) |
| `get_traces` | Get full trace with all observations (generations/spans, token counts) |
| `get_session_summary` | Session-level overview across all 3 trace sources (hooks, proxy, OTEL) |

**Note:** Tracing is fully automatic ‚Äî hooks and proxy middleware handle all ingestion. This server is read-only analytics.

---

### sequential-thinking ‚Äî Reflective Reasoning

A self-contained MCP for multi-step reasoning on design problems. Use it when you need to think through a complex architectural decision before acting.

## 2. Agent Roles

When delegating work via the `Agent` tool, select the appropriate role and model:

| Role | Model | Use For |
|------|-------|---------|
| **Orchestrator** | Opus 4.6 | Task decomposition, delegation (default role) |
| **Architect** | Gemini 3.1 Pro | System design, entity modeling, ADRs |
| **Implementer** | Sonnet 4.6 | Multi-file code changes, tests, refactoring |
| **Reviewer** | Gemini 3 Flash | Code review, checklist validation |
| **Fast-Coder** | Haiku 4.5 | Simple edits, boilerplate, validation code |
| **Researcher** | Gemini 3.1 Pro | Codebase exploration, log analysis |

Each project adds role-specific skills and review checklists via its `<project>-patterns` skill and CLAUDE.md `Project Agent Roles` section.

**Escalation path:** Flash ‚Üí Pro ‚Üí Haiku ‚Üí Sonnet ‚Üí Opus. If a cheaper model fails, escalate to the next tier.

**Delegation decision tree:**
```
Is it a single-step task?
  ‚îú‚îÄ Yes ‚Üí Agent tool with appropriate model
  ‚îî‚îÄ No (multi-step) ‚Üí run_workflow from orchestrator-mcp

Is it a design problem?
  ‚îî‚îÄ Yes ‚Üí sequential-thinking MCP first, then delegate implementation
```

## 3. Skills

Skills are markdown files that provide behavioral guidance to agents. They are loaded automatically based on context.

### Generic skills (in `~/.claude/skills/`)

| Skill | Purpose | Loaded When |
|-------|---------|-------------|
| `agent-orchestrator` | Task decomposition and role selection | Multi-step tasks |
| `tool-router` | Decision tree for optimal tool/model selection | Tool selection decisions |
| `quota-guard` | Always-active cost awareness | Every session |
| `skill-awareness` | Knowledge about the skills ecosystem | Skill-related queries |

### Project skills (in `<project>/.claude/skills/`)

Each project defines two skills:

- **`<project>-patterns`** ‚Äî Domain entities, architecture conventions, API patterns, testing conventions. Loaded during any development work.
- **`<project>-workflows`** ‚Äî Project-specific workflow types (e.g., new-entity, new-page). Loaded when creating new project components.

See the project-specific guide for details on these skills.

## 4. Hooks (Automatic)

Two consolidated hooks run automatically ‚Äî no manual invocation needed:

| Hook | Trigger | Effect |
|------|---------|--------|
| `pre-tool-gate.py` | PreToolUse (`.*`) | Session gate, budget enforcement, model selection, Gemini delegation suggestions |
| `post-tool-trace.py` | PostToolUse (`.*`) | Langfuse tracing, throttle tracking, task/workflow artifact updates, memory queue, doc staleness |

Each hook internally dispatches to multiple handlers. See `observability.md` for the full dispatch table.

Hooks have a 2-5 second timeout and are non-blocking on failure.

### Artifact Skills (User-Invoked)

| Skill | Invocation | Artifact Generated |
|-------|------------|-------------------|
| `implementation-plan` | `/implementation-plan` | `.claude/artifacts/implementation_plan.md` |
| `walkthrough` | `/walkthrough` | `.claude/artifacts/walkthrough.md` |

See `artifacts.md` for the full artifact system documentation.

## 5. Common Workflows

### Starting a New Session
```
1. search_memories("recent work")           ‚Üí Recall context
2. explain_architecture()                    ‚Üí Orient to codebase
3. get_quota_state()                         ‚Üí Check model availability
```

### Code Review
```
1. Get the diff (git diff, PR diff, etc.)
2. If < 50 lines ‚Üí review inline
3. If 50+ lines ‚Üí review_diff("the diff content")
4. Check against project-specific review checklist
```

### After Major Changes
```
1. /walkthrough                              ‚Üí Generate work summary artifact
2. refresh_index()                           ‚Üí Rebuild codebase digest
3. add_memory("Completed: description")      ‚Üí Store in mem0
4. get_cost_report()                         ‚Üí Check spending
```

### Adding a Feature (generic)
```
1. get_quota_state()                         ‚Üí Verify models available
2. /implementation-plan                      ‚Üí Draft plan artifact for review
3. Use sequential-thinking for design if needed
4. run_workflow("feature", "description")    ‚Üí Multi-step execution (generates task_plan.md)
5. /walkthrough                              ‚Üí Generate summary artifact
6. refresh_index()                           ‚Üí Update digest
7. add_memory("Added feature: ...")          ‚Üí Store outcome
```

For project-specific workflows (new-entity, new-page, etc.), see the project guide.

## 6. Project Customization (Part 2)

Each project extends the generic framework with:

1. **`.mcp.json`** ‚Äî MCP server config with project-specific `MEM0_APP_ID` and Langfuse API keys.
2. **`.envrc`** ‚Äî Environment variables (managed by direnv).
3. **`CLAUDE.md`** ‚Äî Project-specific agent roles, review checklists, and development rules.
4. **`.claude/skills/<project>-patterns/`** ‚Äî Domain knowledge skill.
5. **`.claude/skills/<project>-workflows/`** ‚Äî Workflow definitions skill.
6. **`.claude/settings.local.json`** ‚Äî MCP permissions and artifact hooks.
7. **`.claude/hooks/`** ‚Äî Two consolidated hooks (`pre-tool-gate.py`, `post-tool-trace.py`) + shared `lib/`.
8. **`.claude/artifacts/`** ‚Äî Generated deliverables: task plans, implementation plans, reviews, walkthroughs.
9. **`.gemini-index`** ‚Äî Codebase digest (rebuilt via `refresh_index`).
10. **mem0 memories** ‚Äî Seeded entity relationships and project facts.

See `project-setup-guide.md` for step-by-step instructions on creating a new project instance.

## 7. Troubleshooting

| Problem | Solution |
|---------|----------|
| MCP server not connecting | Check `uv run` path and env vars in `.mcp.json` |
| mem0 search returns nothing | Verify `MEM0_APP_ID` matches in `.envrc` and `.mcp.json` |
| Gemini calls failing | Check proxy at `localhost:1338/health` |
| Langfuse empty | Verify API keys in `.mcp.json`, check hook is registered |
| Neo4j Cypher errors | Use underscores not hyphens in relationship names |
| PointStruct validation errors | Stale entries with null vectors ‚Äî non-blocking, ignore or prune |
| "ThinkingBlock" crash in mem0 | Monkey-patch is applied at server startup ‚Äî restart the MCP server |
| Ollama not reachable | Start Ollama (`ollama serve`) ‚Äî needed for mem0 embeddings |
